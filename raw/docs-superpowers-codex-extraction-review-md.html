    Codex extraction review (2026-05-01)

    TL;DR

    • The pipeline does not extract the ontology it advertises. Algorithm, Benchmark, Result, Person, Organization, Metric, Model, Capability, ArchitecturePattern, TrainingParadigm, InferenceStrategy, ContributionClaim, ComparisonClaim, and OpenQuestion are all zero in the live graph because the extractor has no rules for them.
    • The paper-title gate is not a title gate. It accepts code fences and assistant/provenance chatter as paper_file, so public paper pages and syntheses contain titles like ````markdown and 번역 완료. 파일 생성됨: ...`.
    • The controlled-vocabulary claim is only half true. Term extraction is controlled, but heading extraction mints arbitrary Concept nodes from docs, including route names like /graph.json and design-section headings.
    • Repositories and papers are not semantically connected. Repo files carry arxiv_id metadata, but there is no implemented_in edge between the paper and repository; this wrecks the paper/repo layer the site claims to expose.
    • The public-site filters hide some damage, but .llm-wiki/graph.json is still a bloated mixed graph: 1362 of 1838 nodes are code graph nodes, and many claim/evidence nodes are considered "public" by the core public predicate even though no wiki route renders them.

    Findings

    F-1: The title validator promotes scraper and assistant garbage to public Papers

    • Severity: critical.
    • Where: llm_wiki/research_graph.py:958, llm_wiki/research_graph.py:998, llm_wiki/research_graph.py:1030, llm_wiki/wiki_projector.py:92.
    • What's wrong: extract_title() accepts the first line that passes looks_like_research_title(), and that predicate only checks length, a few prefixes, and "has a letter". That is not enough for production. is_verified_paper_title() then upgrades that line to title_quality="paper_file", and WikiLayerProjector.project() publishes it because is_public_research_node() trusts that quality.
    • Evidence: live .llm-wiki/graph.json has public Paper nodes named:
    • ````markdown from data/research/daily/2026-04-23/papers/2604.11251/paper.md`; the raw file has the code fence at line 7 and the real title at line 16.
    • 번역 완료. 파일 생성됨: \data/daily/2026-04-06/papers/2304.12210/paper_ko.md\` from data/research/daily/2026-04-06/papers/2304.12210/paper.md`; the raw file is only translation status text at lines 9-11.
    • 실제 중국어 논문 분석이 들어있는 파일을 확인해야 합니다. \prompt.txt\ 파일을 확인해보겠습니다. from data/research/daily/2026-04-07/papers/2410.17897/paper.md; raw lines 7-9 are extractor/assistant chatter, not a paper.
    • Fix: replace looks_like_research_title() with a stricter classify_paper_title_candidate(line, context) that rejects code fences, markdown file paths, "파일", "확인", "번역 완료", "생성됨", assistant/action verbs, bare prose sentences, and lines inside fenced blocks. For paper files, prefer explicit patterns in order: # <title> after ## #1, <title> | Cool Papers, arXiv metadata title if present, then arXiv placeholder. Add corpus regression tests using the three exact raw files above.

    F-2: title_quality is a four-value enum in name only

    • Severity: major.
    • Where: llm_wiki/research_graph.py:162, llm_wiki/research_graph.py:340, llm_wiki/research_graph.py:468, llm_wiki/research_graph.py:649, llm_wiki/synthesis.py:902.
    • What's wrong: The rank table includes reference_context, but _extract_paper_references() never sets it and infer_arxiv_reference_title() is dead code. The current extractor only emits paper_file and arxiv_only, and paper_file is polluted by F-1. Synthesis then hard-gates on {paper_file, verified}, which means bad titles get synthesized while valid digest-only titles can never surface.
    • Evidence: rg finds infer_arxiv_reference_title only at its definition. Live graph has 80 Paper nodes, 40 public by title quality, and at least three of those public nodes are not paper titles. The site search index includes title: "`markdown" and title: "번역 완료. 파일 생성됨: ..." from those nodes.
    • Fix: make title_quality an explicit enum and require every Paper path to set one of verified, paper_file, reference_context, arxiv_only, invalid. Wire infer_arxiv_reference_title() into digest/feed extraction only if source context is a structured arXiv card, and keep reference_context non-public until resolved or manually verified. Treat failed paper files as invalid, not arxiv_only, so broken inputs are visible in diagnostics without becoming papers.

    F-3: False negatives are baked into paper-file title extraction

    • Severity: major.
    • Where: llm_wiki/research_graph.py:958, llm_wiki/research_graph.py:998, tests/test_research_corpus.py:46.
    • What's wrong: A paper file without an explicit title is intentionally downgraded to arxiv_only, even when the body clearly contains a paper abstract and named method. The test at tests/test_research_corpus.py:46 locks in this false negative: 2604.02996 becomes arXiv:2604.02996 because the title is missing from the scrape. That hides a real paper from public pages because is_public_research_node() rejects arxiv_only.
    • Evidence: data/research/daily/2026-04-07/papers/2604.02996/paper.md:7 describes "MHMO(Multi-Human Multi-Object) 렌더링" and a "3D Gaussian Splatting 기반의 새로운 계층적 프레임워크인 MM-GS". The live graph still contains only Paper arXiv:2604.02996, title_quality=arxiv_only, so the real paper does not get a public page.
    • Fix: add a resolver stage for paper.md files with valid arxiv_id but no title. Structure:
    • resolve_missing_paper_title(arxiv_id: str, text: str, metadata: dict) -> ResolvedTitle
    • first parse known local scrape fields, then use an offline arXiv metadata cache, then optionally mark as needs_metadata.
    • never infer a title from the abstract body unless a pattern has high precision; emit a diagnostics record instead.

    F-4: The advertised ontology is mostly unreachable

    • Severity: critical.
    • Where: llm_wiki/research_graph.py:20, llm_wiki/research_graph.py:287, llm_wiki/research_graph.py:373, llm_wiki/research_graph.py:947.
    • What's wrong: The enum lists a rich ontology, but extraction only has 18 hard-coded TermRules plus a crude claim classifier. There are no extraction paths for Person, Organization, Model, Algorithm, Benchmark, Metric, Result, ArchitecturePattern, TrainingParadigm, InferenceStrategy, Capability, ContributionClaim, ComparisonClaim, or OpenQuestion. This is not a corpus problem; the corpus contains authors, organizations, datasets, benchmarks, metrics, and results.
    • Evidence: live graph counts are zero for all the types above. Raw files contain missed entities: data/research/daily/2026-04-25/papers/2604.15941/paper.md:26 mentions Mip-NeRF360 and High-Frequency datasets/benchmarks; data/research/daily/2026-04-29/papers/2604.23537/paper.md:28 mentions DTU and Tanks and Temples; many paper files have 저자: or Authors: blocks. None become Dataset, Benchmark, Person, Metric, or Result.
    • Fix: implement typed extractors instead of adding more TermRules:
    • extract_authors(text) -> list[Person] from Authors: / 저자: blocks with authored_by edges.
    • extract_eval_entities(text) -> datasets, benchmarks, metrics, results using a curated registry plus numeric-result patterns.
    • extract_method_entities(text) -> algorithms, models, training_paradigms, inference_strategies.
    • Keep each extractor precision-first and backed by fixture tests from the real files above.

    F-5: Controlled vocabulary is violated by heading-derived Concepts

    • Severity: major.
    • Where: llm_wiki/research_graph.py:403, llm_wiki/research_graph.py:423, llm_wiki/research_graph.py:901.
    • What's wrong: The spec promises a controlled ontology to avoid noun-phrase pollution, but _add_document_structure() still creates Concept nodes from arbitrary headings when _is_concept_shaped_heading() sees a whitelisted token. That turns docs and design routes into concepts. The cap of six headings per file limits blast radius; it does not make the nodes valid.
    • Evidence: live graph has Concept nodes such as /graph/ ✅, /graph.json ✅, /graph.jsonld ✅, 4.1 Synthesis layers, The Karpathy three-layer model, and Dependency graph, all with metadata={"source_kind": "document_heading"}. These are not research concepts in the same ontology as Gaussian Splatting or Depth Map.
    • Fix: stop minting Concept from headings. Emit a separate non-public Section or DocumentSection type, or store headings in source-document metadata. Only create Concept when the heading exactly matches a registered term or an approved alias; do not use token-level whitelist matching.

    F-6: Concept / TechnicalTerm / MethodologicalConcept / Algorithm dispatch is not principled

    • Severity: major.
    • Where: llm_wiki/research_graph.py:287, llm_wiki/research_graph.py:931.
    • What's wrong: Type assignment is whatever the author of DEFAULT_TERM_RULES typed that day. Gaussian Splatting is MethodologicalConcept, Depth Map is TechnicalTerm, Multi-View Consistency is EvaluationProtocol, but no rule can ever create Algorithm, ArchitecturePattern, TrainingParadigm, or InferenceStrategy. relation_for_node_type() then maps most types to generic uses, which hides ontology mistakes rather than exposing them.
    • Evidence: live graph has 4 MethodologicalConcept, 2 TechnicalTerm, 1 EvaluationProtocol, and 0 Algorithm. Raw paper text says "GaussianFlow SLAM을 제안한다" and "SDFRaster를 제안한다", but those method names are not extracted as Algorithm; they only survive as part of paper titles.
    • Fix: introduce a typed term registry with required fields: canonical_name, node_type, aliases, match_scope, relation, public, owner. Reject registry entries where node_type falls through to generic uses unless explicitly allowed. Add tests asserting at least one real Algorithm extraction from GaussianFlow SLAM or SDFRaster.

    F-7: Repositories are extracted as orphan source artifacts

    • Severity: major.
    • Where: llm_wiki/research_graph.py:777, llm_wiki/research_graph.py:364, llm_wiki/research_graph.py:442, llm_wiki/project.py:167.
    • What's wrong: source_kind_to_node_type() correctly classifies papers/<id>/repo.md as Repository, and extract_source_metadata() captures its arxiv_id. Then nothing connects it to the corresponding paper. The graph has implemented_in in the ontology, but the extractor never emits it for paper/repo pairs.
    • Evidence: live graph has Paper RAVEN... with arxiv_id=2509.23563 and Repository GitHub 분석: facebookresearch/map-anything also associated with that arXiv id. They coexist, but the edge list does not encode Paper -> implemented_in -> Repository. The wiki pages therefore rely on incidental shared topics, not the actual paper-code relationship.
    • Fix: when extracting a Repository with metadata["arxiv_id"], add or upsert a placeholder Paper with id_seed=f"arXiv:{arxiv_id}", then add paper --implemented_in--> repository. If the paper file later arrives, prefer_research_node() can upgrade the paper title while preserving the edge.

    F-8: Repository identity is based on display text, not URL

    • Severity: major.
    • Where: llm_wiki/research_graph.py:223, llm_wiki/research_graph.py:768, llm_wiki/research_graph.py:1039.
    • What's wrong: Paper identity has id_seed=f"arXiv:{id}"; repository identity does not get the equivalent id_seed=github_url. The node id is derived from the title extracted from the markdown, usually GitHub 분석: owner/repo. A title wording change, Korean/English prefix change, or mirrored weekly repo note can split the same repository into multiple ids.
    • Evidence: live graph names are presentation strings like GitHub 분석: Tencent-Hunyuan/HY-WorldPlay, GitHub 분석: facebookresearch/map-anything, and GitHub 분석: SkalskiP/top-cvpr-2026-papers. Raw repo files contain stable URLs, e.g. data/research/weekly/2026-W15/repos.md:250 has https://github.com/Tencent-Hunyuan/HY-WorldPlay, but that URL is not used as the identity seed.
    • Fix: parse repository URLs in extract_source_metadata() (github.com/<owner>/<repo> normalized to lower-case owner/repo, stripping .git). In extract_text(), if source_type == REPOSITORY and github_repo, call builder.add_node(display_name, REPOSITORY, id_seed=f"github:{owner}/{repo}", aliases=[owner/repo, url]).

    F-9: CodeProject and Repository are semantically overlapping but operationally unrelated

    • Severity: major.
    • Where: llm_wiki/research_graph.py:31, llm_wiki/research_graph.py:40, llm_wiki/project.py:167, llm_wiki/project.py:176, llm_wiki/code_graph.py:22, llm_wiki/wiki_projector.py:28.
    • What's wrong: Repository means external GitHub repo notes; CodeProject means the current local workspace. Both project into repos, but only CodeProject triggers code-graph extraction. The disambiguation is hidden in ProjectWiki.ingest(): markdown from a repository/code project is forcibly ingested as SourceDocument, then code graph is merged separately.
    • Evidence: live graph has 116 Repository nodes and exactly 1 CodeProject (LLM-Wiki). The site Repos index mixes external research repos with the local project node because _KIND_FOR_TYPE maps both to repos.
    • Fix: rename or separate the kinds: ExternalRepository for GitHub artifacts, LocalCodeProject for code graphs. If enum churn is too large, add metadata["repository_scope"] in {"external", "local"} and render separate routes/facets. Do not map local code project into the same index without a visible discriminator.

    F-10: is_public_research_node() is not the public contract its name implies

    • Severity: major.
    • Where: llm_wiki/research_graph.py:658, llm_wiki/wiki_projector.py:92, llm_wiki/site/search.py:53.
    • What's wrong: The function only hides social feeds and unverified papers. It returns true for CodeFunction, SourceFile, Dependency, EvidenceSpan, and claim nodes. Projection and search have their own second-layer allowlists, so "public" means different things in different modules.
    • Evidence: applying the function logic to live graph leaves 1798 of 1838 nodes "public", including 1122 CodeFunction, 83 SourceFile, 78 EvidenceSpan, and 77 Dependency nodes. site/search.py:96 documents that these are excluded from URLs/search, proving the core predicate is too broad.
    • Fix: split the predicate into is_valid_public_entity(node) and is_exposed_wiki_page(node). Put the wiki-layer allowlist in one shared module and make projector, search, pages, JSON export, and synthesis all call that. Add a test that CodeFunction, EvidenceSpan, and Claim are not public wiki nodes.

    F-11: Code graph nodes bloat the research graph and should not share the same artifact by default

    • Severity: major.
    • Where: llm_wiki/project.py:176, llm_wiki/code_graph.py:26, llm_wiki/code_graph.py:79.
    • What's wrong: The code graph is dumped into the same ResearchGraph as research entities. It contributes 1363 nodes (CodeProject, SourceFile, CodeClass, CodeFunction, Dependency) out of 1838 total, about 74% of the graph. Hiding those nodes from the website does not fix graph size, MCP noise, export noise, or ranking noise for consumers of .llm-wiki/graph.json.
    • Evidence: live counts: CodeFunction=1122, CodeClass=80, SourceFile=83, Dependency=77, CodeProject=1. That is far larger than all paper, repository, source, concept, and synthesis nodes combined.
    • Fix: write code graph to .llm-wiki/code-graph.json and keep .llm-wiki/graph.json research/wiki-layer first. If consumers need a union graph, generate .llm-wiki/combined-graph.json explicitly. At minimum add --include-code-graph defaulting off for project compile on research corpora.

    F-12: Source-document handling still leaks docs and weekly headings into Concepts

    • Severity: minor.
    • Where: llm_wiki/research_graph.py:337, llm_wiki/research_graph.py:403, llm_wiki/research_graph.py:410, llm_wiki/research_graph.py:418.
    • What's wrong: Recent changes stopped many daily-digest paper headings from becoming concepts, but the rule still leaks Korean/mixed headings and numeric-prefixed design sections when they contain whitelist tokens. _strip_heading_numbering() plus token whitelist makes headings like 4.1 Synthesis layers look like concepts. This is especially bad because docs under docs/ are compiled alongside research data.
    • Evidence: live Concept nodes include 4.1 Synthesis layers from docs/superpowers/specs/2026-04-27-wiki-frontend-redesign-design.md, Geometry-Grounded Gaussian Splatting from data/research/weekly/2026-W17/papers.md, and route names from docs/frontend-redesign.md. These are section headings, not extracted research entities.
    • Fix: for SourceDocument, make heading extraction structural only. Store headings under metadata["sections"] with normalized anchors. Promote to ontology nodes only through the typed registry or an explicit frontmatter field like concepts: [...].

    F-13: Claim extraction cannot create the claim types the ontology promises

    • Severity: major.
    • Where: llm_wiki/research_graph.py:389, llm_wiki/research_graph.py:498, llm_wiki/research_graph.py:509, llm_wiki/research_graph.py:947.
    • What's wrong: classify_claim_type() can only emit PerformanceClaim, LimitationClaim, CausalClaim, or generic Claim. ContributionClaim, ComparisonClaim, and OpenQuestion are dead enum values. _add_comparative_claims() also emits PerformanceClaim, not ComparisonClaim, even when a sentence is explicitly comparative.
    • Evidence: live graph has PerformanceClaim=54, Claim=20, CausalClaim=6, LimitationClaim=5, and zero ContributionClaim, ComparisonClaim, OpenQuestion. Raw data/research/daily/2026-04-29/papers/2604.23537/paper.md:28 says SDFRaster outperforms latest methods on DTU and Tanks and Temples; that should produce a comparison/result shape, not just a generic performance blob.
    • Fix: separate claim extraction from term evidence. Add:
    • extract_contribution_claims(text) for "we propose/present/introduce" and Korean equivalents.
    • extract_comparison_claims(text) for "outperforms/보다/compared with" plus baseline extraction.
    • extract_open_questions(text) for explicit "future work/open question/remains unclear" patterns.
    • Keep PerformanceClaim for numeric/score claims and link it to Result, Metric, and Benchmark when present.

    F-14: Tests pin tiny happy paths and miss live-corpus regressions

    • Severity: major.
    • Where: tests/test_research_corpus.py:32, tests/test_research_corpus.py:244, tests/test_extraction_quality.py:171, tests/test_project_cli.py:161.
    • What's wrong: The tests contain hand-built snippets for recent fixes, but they do not run against the real broken files now in data/research. Worse, one test explicitly expects a real paper to stay hidden as arxiv_only. There is no test asserting ontology coverage for Person, Benchmark, Result, Algorithm, or paper/repo edges.
    • Evidence: tests/test_research_corpus.py:244 blocks one scaffold sentence, but the live graph still promotes a different scaffold sentence from the same raw file. tests/test_extraction_quality.py:171 only asserts papers > concepts on a small fixture, while the real graph still has route names and spec headings as concepts.
    • Fix: add a corpus-quality test suite over selected real files:
    • bad title blacklist: 2304.12210, 2410.17897, 2604.11251.
    • missing title diagnostics: 2604.02996.
    • positive extraction: authors from 2604.15612, benchmarks/datasets from 2604.15941 and 2604.23537, repo edge for 2509.23563.
    • graph-level invariant: no public paper title starts with a fence, contains .md, or contains Korean assistant/provenance phrases.

    Recommendations

    • S: Kill public bad paper titles first. Tighten looks_like_research_title()/extract_title() and add blacklist tests for the three live failures. This is the fastest visible quality win.
    • S: Add repo URL identity and paper-repo edges. Parse GitHub URLs, seed repository IDs by URL, and emit implemented_in edges from arxiv_id metadata.
    • S: Stop heading-derived Concept pollution. Convert arbitrary headings to source-document sections; only promote exact registry matches.
    • M: Centralize public/wiki-layer filtering. One predicate should drive wiki projection, search, pages, JSON exports, and synthesis inputs.
    • M: Split code graph output from research graph output. Keep .llm-wiki/graph.json usable for research/wiki consumers; generate a combined graph only when requested.
    • M: Build typed extractors for authors and evaluation entities. Start with Person, Dataset, Benchmark, Metric, and Result because the corpus already exposes those in regular patterns.
    • M: Replace DEFAULT_TERM_RULES with a typed registry. Require node type, relation, aliases, and tests for each term. Add at least one real Algorithm.
    • L: Add an offline metadata resolver for arXiv IDs. This fixes missing titles without hallucinating from abstracts and gives title_quality real semantics.
    • L: Rework claim extraction into contribution/comparison/result/open-question stages. The current sentence keyword classifier is not enough for the assertion layer.

    Out of scope but worth noting

    The synthesis layer is downstream of bad extraction and currently amplifies it. Live wiki output includes daily synthesis links such as `[ ``markdown (arXiv:2604.11251)]`` and Korean assistant-status strings as paper titles. Even if extraction is fixed, synthesis should defend itself with its own title sanity check before writing public pages.

    The site surface claims clean categories, but the graph payload still ships a mixed internal graph. That is fine for a debugging endpoint, but it is not fine as the canonical artifact agents consume. The project needs separate artifact contracts: research graph, code graph, public wiki graph, and optional debug union.
